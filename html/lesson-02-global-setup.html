الدرس الثاني: إعداد النظام عالمياً

كيفية تثبيت وتهيئة Everything Claude Code للعمل عبر كافة مشاريعك، مع أمثلة حقيقية.

حالة استخدام: التثبيت للمشاريع الكبيرة مقابل الصغيرة

يوفر نظام ECC مسارات تثبيت مختلفة حسب حجم مشروعك واحتياجاتك.

المثال الأول: تثبيت سريع لمشروع صغير عبر إضافة Claude

إذا كنت تعمل على سكربت بسيط أو موقع شخصي، لا تحتاج للتعقيد. استخدم هذا في طرفية Claude Code:

# أضف السوق الرسمي للإضافة
/plugin marketplace add https://github.com/affaan-m/ECC

# قم بتثبيت الإضافة
/plugin install ecc@ecc

المثال الثاني: تثبيت دقيق لمشروع ضخم (بدون خطافات مزعجة)

في مشاريع Enterprise الكبيرة، قد لا ترغب بأن تقوم الخطافات بمقاطعتك باستمرار. يمكنك استخدام التثبيت اليدوي مع الملف الشخصي "الأساسي" (Minimal) الذي يركز على الوكلاء والقواعد فقط:

# في طرفية Bash أو PowerShell
./install.sh --profile minimal --target claude

حالة استخدام: إعداد خوادم MCP بشكل عملي

خوادم MCP تستهلك مساحة كبيرة من السياق. لنفترض أنك تعمل على مشروع يحتاج للاتصال بقاعدة بيانات PostgreSQL مباشرة.

بدلاً من تفعيل جميع الخوادم الممكنة، قم بإنشاء ملف .mcp.json في مجلد مشروعك يحتوي فقط على الخادم الذي تحتاجه:

{
  "postgres": {
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-postgres", "postgresql://user:password@localhost:5432/mydb"]
  }
}

لماذا هذا مهم عملياً؟ إذا قمت بتفعيل 15 خادم MCP عالمياً في إعداداتك (~/.claude/settings.json)، فإن نافذة الـ 200k رموز الخاصة بك ستنكمش إلى حوالي 70k فوراً، مما يجعلك تدفع تكاليف أعلى وتفقد الذاكرة بسرعة. التفعيل المحلي هو الحل.

حالة استخدام متقدمة: التبديل الاستراتيجي بين النماذج (Models) أثناء بناء تطبيق

للتحكم في التكاليف وسرعة الإنجاز بشكل احترافي، المطورون المحترفون لا يستخدمون أذكى (وأغلى) نموذج طوال الوقت. في Claude Code (عبر ECC)، يمكنك التبديل بين النماذج بمرونة حسب المهمة.

1. إعداد المشروع للتبديل السريع

تأكد أن ملف الإعدادات لديك يدعم النموذج الافتراضي المناسب مع تفعيل الـ Thinking Tokens إذا لزم الأمر:

// في مسار ~/.claude/settings.json
{
  "model": "claude-3-7-sonnet-20250219",
  "env": {
    "MAX_THINKING_TOKENS": "10000"
  }
}

2. سيناريو بناء تطبيق (مثال: تطبيق إدارة مهام TeamTask)

كيف نوزع الجهد على النماذج المختلفة أثناء دورة حياة بناء هذا التطبيق؟

المرحلة الأولى: التخطيط المعماري المعقد (استخدام Opus أو النماذج المعمقة)

عندما تحتاج لتصميم بنية قاعدة البيانات، أو هندسة نظام صلاحيات معقد للمستخدمين (Role-based access)، أنت تحتاج لنموذج يتميز بالمنطق العميق ليتفادى الأخطاء الجوهرية.

# التبديل للنموذج الأذكى (مثلاً Opus)
/model claude-3-opus-20240229

# ثم نطلب التخطيط الاستراتيجي
/ecc:plan "صمم بنية قاعدة بيانات لتطبيق إدارة مهام يدعم فرق العمل، بصلاحيات دقيقة (Admin, Editor, Viewer). اكتب الـ Schema بصيغة Prisma وضع في الحسبان قابلية التوسع."

المرحلة الثانية: كتابة الكود اليومي و الـ TDD (استخدام Sonnet)

أغلب عملية التطوير اليومية (إنشاء مسارات API، كتابة مكونات React، كتابة الاختبارات) لا تحتاج لنموذج مكلف جداً كـ Opus. نموذج Sonnet 3.7 هو الخيار الذهبي (Workhorse) هنا من حيث توازن السرعة والجودة.

# العودة للنموذج السريع والذكي برمجياً
/model claude-3-7-sonnet-20250219

# استدعاء مهارة TDD لكتابة الميزة بناءً على الخطة
tdd-workflow skill "بناءً على خطة Prisma، قم بإنشاء مسار API لإضافة مهمة جديدة مع التأكد من صلاحيات المستخدم (Role)."

المرحلة الثالثة: المهام الروتينية السريعة (استخدام Haiku)

أحياناً تريد فقط تنسيق الملفات، كتابة تعليقات (Docstrings)، تحويل تصميم CSS عادي إلى Tailwind، أو تلخيص ملفات Log ضخمة. هذه المهام الروتينية يمكنك رميها على نموذج Haiku السريع جداً والرخيص للغاية لتوفير ميزانيتك.

# التبديل للنموذج الأسرع والأرخص
/model claude-3-5-haiku-20241022

# إعطاء أمر روتيني لا يحتاج ذكاءً خارقاً
claude "قم بالمرور على جميع ملفات المجلد /components وأضف تعليقات JSDoc تشرح وظيفة كل مكون و الـ Props الخاصة به."

الخلاصة العملية للتبديل:

  • Opus / النماذج العميقة: لحل المشاكل المعمارية، وإصلاح أخطاء الـ Production الغامضة (البحث عن إبرة في كومة قش). استخدامه يمثل 10% من وقتك.
  • Sonnet: الحصان الدؤوب. 80% من وقتك سيكون هنا (كتابة ميزات، TDD، مراجعة الكود الأمنية).
  • Haiku: للمهام المتكررة وتنسيق البيانات وقراءة السجلات الطويلة جداً. استخدامه يمثل 10% من وقتك لترشيد الاستهلاك.
