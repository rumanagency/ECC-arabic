**نظام التشغيل الخاص بحامل الوكيل.**

ليس مجرد إعداد. نظام كامل: المهارات، الغرائز، تحسين الذاكرة، التعلم المستمر، فحص الأمان، والبحث أولاً. جاهز للإنتاج. مبني من واقع إنشاء تدفقات عمل وكيلية متعددة الحوامل.

يعمل عبر **Codex**، **Claude Code**، **Cursor**، **OpenCode**، **Gemini**، **Zed**، **GitHub Copilot**، ومواضع وكيلية أخرى.

توفر نسخة `ECC v2.0.0` قصة مشغل Hermes العامة فوق هذه الطبقة القابلة لإعادة الاستخدام: ابدأ بـ [دليل إعداد Hermes](docs/HERMES-SETUP.md)، ثم راجع [ملاحظات الإصدار 2.0.0](docs/releases/2.0.0/release-notes.md) و[العمارة متعددة الحوامل](docs/architecture/cross-harness.md).

## ما الجديد

### v2.0.0 — نظام تشغيل حامل الوكلاء (يونيو 2026)

تخرُّج ثابت لخط 2.0: 261 مهارة، الركيزة اللوحية الضابطة (محولات الجلسة + جرد MCP)، خدمة دورة حياة worktree، عائلة منظمي `orch-*`، وإطلاق مجتمع مشغل Hermes العام. الملاحظات الكاملة: [docs/releases/2.0.0/release-notes.md](docs/releases/2.0.0/release-notes.md).

### v2.0.0-rc.1 — تحديث السطح، تدفقات العمل المشغل، وألفا ECC 2.0 (أبريل 2026)

- **واجهة داشبورد** — تطبيق جديد قائم على Tkinter (`ecc_dashboard.py` أو `npm run dashboard`) مع تبديل الوضع الداكن/الفاتح، تخصيص الخط، وشعار المشروع في الرأس وشريط المهام.
- **السطح العام المتزامن مع المستودع المباشر** — بيانات وصفية، تعداد الكتالوج، وملفات التكوين تتطابق مع السطح المفتوح: 64 وكيلًا، 262 مهارة، و84 شيم أمر متوارٍ.
- **قصة المشغل العام** — `Hermes setup guide` فوق طبقة إعادة الاستخدام.
- **نفق التشغيل والتحكم** — `session adapters`، جرد أدوات MCP، وتجربة تركيبية جديدة.
- **التوسيع الخارجية** — `brand-voice`، `social-graph-ranker`، `connections-optimizer`، `customer-billing-ops`، `ecc-tools-cost-audit`، `google-workspace-ops`، `project-flow-ops`، و `workspace-surface-audit` تكمل خط المشغل.
- **أدوات الوسائط والإطلاق** — `manim-video`، `remotion-video-creation`، وترقيات النشر الاجتماعي تجعل الشروحات التقنية والأدوات الإعلامية جزءًا من نفس النظام.
- **نموذج المنتج والنظام الأساسي** — `nestjs-patterns`، تغطية التثبيت لـ Codex/OpenCode، وتوسيع التغليف عبر الحوامل.
- **نموذج “الطائرة التنبؤية” Itô** — `ito-market-intelligence`، `ito-basket-compare`، `ito-trade-planner`، `ito-data-atlas-agent`، `prediction-market-oracle-research`، و `prediction-market-risk-review` تضيف تدفقات عمل عامة غير استشارية مع إبقاء الوصول إلى واجهة Itô الآمنة منفصلة عن الفوترة.
- **مهارة التحسين** — `parallel-execution-optimizer`، `benchmark-optimization-loop`، `data-throughput-accelerator`، `latency-critical-systems`، و `recursive-decision-ledger` تحول التكرار إلى حزم محدودة من المراقبة والأداء.
- **ECC 2.0 ألفا في الشجرة** — النموذج الأولي للوحة التحكم بلغة Rust في `ecc2/` يبني محليًا ويعرض `dashboard`، `start`، `sessions`، `status`، `stop`، `resume`، و `daemon`.
- **لقطات حالة المشغل** — `ecc status --markdown --write status.md` يحول حالة المتجر المحلي إلى تسليم محمول يغطي الجهوزية، الجلسات النشطة، صحة مهام المهارة، صحة التثبيت، والأشياء المعلقة من Linear/GitHub/التسليم اليدوي. استخدم `ecc work-items upsert ...` لإدخالات يدوية، `ecc work-items sync-github --repo owner/repo` لقائمة PR/issue، و `ecc status --exit-code` لفشل الأتمتة عند الحاجة.
- **التدقيق الشامل** — AgentShield، ضوابط تكلفة ECC Tools، بوابة الفوترة، وتجديد الموقع تستمر في الشحن حول النواة بدلًا من التشظي.

### v1.9.0 — التثبيت الانتقائي وتوسيع اللغة (مارس 2026)

- **هندسة التثبيت الانتقائي** — أنبوب التثبيت المعتمد على البيان مع `install-plan.js` و `install-apply.js` للتثبيت المستهدف. يقوم سجل الحالة بتتبع ما تم تثبيته ويمكّن التحديثات التدريجية.
- **6 وكلاء جدد** — `typescript-reviewer`، `pytorch-build-resolver`، `java-build-resolver`، `java-reviewer`، `kotlin-reviewer`، `kotlin-build-resolver` توسع تغطية اللغة إلى 10 لغات.
- **مهارات جديدة** — `pytorch-patterns` لتدفقات عمل التعلم العميق، `documentation-lookup` للبحث المرجعي في API، `bun-runtime` و `nextjs-turbopack` لأدوات JS الحديثة، بالإضافة إلى 8 مهارات تشغيلية و `mcp-server-patterns`.
- **البنية التحتية للجلسات والحالة** — مخزن حالة SQLite مع CLI استعلام، محولات الجلسات للتسجيل المهيكل، أسس تطور المهارة للتعلم الذاتي.
- **إصلاح التوافق** — جعل درجة تدقيق الحامل أكثر حتمية، تعليق المراقبة، وإصلاح موثوقية التشغيل.
- **توسيع الحوامل** — قواعد جديدة للغة Java، PHP، Perl، Kotlin/Android/KMP، C++، وRust تنضم إلى TypeScript وPython وGo وcommon.
- **مساهمات المجتمع** — ترجمات كورية وصينية، تحسينات خطاف السحابة، مهارات معالجة الفيديو، مهارات تشغيلية، مثبت PowerShell، ودعم Antigravity IDE.
- **تعزيز CI** — 19 إصلاحًا لفشل الاختبار، فرض تعداد الكتالوج، تحقق بيان التثبيت، ومجموعة الاختبار الكاملة الخضراء.

### v1.8.0 — نظام أداء الحامل (مارس 2026)

- **إصدار التركيز على الحامل** — أصبحت ECC الآن مؤطرة صراحة كنظام أداء حامل الوكلاء، وليس مجرد حزمة إعداد.
- **إصلاح موثوقية الخطافات** — التجاوز الاحتياطي لـ SessionStart، ملخَّصات مرحلة التوقف، وسكربتات Node.js لتحل محل الأوامر المضمنة الهشة.
- **ضوابط وقت تشغيل الخطاف** — `ECC_HOOK_PROFILE=minimal|standard|strict` و `ECC_DISABLED_HOOKS=...` للتعديل بدون تحرير الملفات.
- **أوامر وسط جديدة** — `/harness-audit`، `/loop-start`، `/loop-status`، `/quality-gate`، `/model-route`.
- **NanoClaw v2** — توجيه النموذج، التحميل الساخن للمهارات، وتخفيف انفجار السياق.
- **تكافؤ عبر الحوامل** — تم تشديد السلوك عبر Claude Code وCursor وOpenCode وCodex.
- **997 اختبار داخلي ناجح** — مجموعة كاملة خضراء بعد إعادة تصميم الخطاف/وقت التشغيل والتوافق.

### v1.7.0 — التوسع عبر الأنظمة الأساسية ومنشئ العرض التقديمي (فبراير 2026)

- **توسيع تغطية الأدوات** — دعم Cursor وCodex وOpenCode أحكم حتى يصبح نفس المستودع قابلاً للشحن عبر معظم الحوامل الكبرى.
- **992 اختبار داخلي** — توسيع التحقق والتغطية عبر الوكلاء والمهارات والأوامر والتغليف.

يجب على معظم مستخدمي Claude Code أن يستخدموا مسار التثبيت واحدًا فقط:

- **الافتراضي الموصى به:** ثبّت الإضافة Claude Code، ثم انسخ مجلدات القواعد التي تحتاجها فقط.
- **استخدم المثبت اليدوي فقط إذا** كنت تريد تحكمًا أدق بما يتم تثبيته، أو تريد تجنب مسار الإضافة بالكامل، أو إذا كان إصدار Claude Code لديك يواجه صعوبة في حل الإدخال السوقي المستضاف ذاتيًا.
- **لا تكدس طرق التثبيت.** أكثر إعدادات التعطل شيوعًا هي: تثبيت الإضافة أولًا، ثم تشغيل `install.sh --profile full` أو `npx ecc-install --profile full` لاحقًا.

إذا قمت بطبقات تثبيت متعددة وظهرت تكرارات، فانطلق مباشرة إلى [إعادة التعيين / إلغاء تثبيت ECC](#reset--uninstall-ecc).

### مسار منخفض السياق / بدون خطافات

إذا كانت الخطافات تبدو شاملة جدًا أو تريد فقط قواعد ECC، وكلاءه، وأوامره، فاختر ملف التعريف اليدوي البسيط:

```bash
./install.sh --profile minimal --target claude
```

```powershell
.\install.ps1 --profile minimal --target claude
# أو
npx ecc-install --profile minimal --target claude
```

هذا الملف يهدف عمدًا إلى استبعاد `hooks-runtime`.

إذا كنت تريد الملف الشخصي الأساسي العادي لكن تحتاج إلى تعطيل الخطافات، استخدم:

```bash
./install.sh --profile core --without baseline:hooks --target claude
```

أضف الخطافات لاحقًا فقط إذا كنت تريد تنفيذًا وقت التشغيل:

```bash
./install.sh --target claude --modules hooks-runtime
```

### جد المكونات الصحيحة أولاً

إذا كنت غير متأكد من الملف الشخصي أو المكونات التي تحتاجها، اطلب المستشار المرفق من أي مشروع:

```bash
npx ecc consult "security reviews" --target claude
```

سيُرجع المكونات المطابقة، ملفات التعريف المرتبطة، وأوامر المعاينة/التثبيت. استخدم أمر المعاينة قبل التثبيت إذا كنت تريد فحص خطة الملف بدقة.

للأعمال ML/MLOps الإنتاجية، احتفظ بالتثبيت اختياريًا ومقتصرًا على المكون:

```bash
npx ecc consult "mlops training model deployment" --target claude
npx ecc install --profile minimal --target claude --with capability:machine-learning
```

### الخطوة 1: تثبيت الإضافة (الموصى به)

> ملاحظة: الإضافة مريحة، لكن المثبت الحر لا يزال المسار الأكثر موثوقية إذا كان إصدار Claude Code لديك يواجه مشكلات في حل إدخال السوق المستضاف ذاتيًا.

```bash
# أضف السوق
/plugin marketplace add https://github.com/affaan-m/ECC

# ثبّت الإضافة
/plugin install ecc@ecc
```

### التسمية + ملاحظة الهجرة

أصبح لدى ECC الآن ثلاثة معرفات عامة، وليست قابلة للتبديل:

- مستودع المصدر على GitHub: `affaan-m/ECC`
- معرف الإضافة/السوق في Claude: `ecc@ecc`
- حزمة npm: `ecc-universal`

هذا مقصود. تُفهرس تثبيتات سوق Anthropic بواسطة معرف إضافة معياري، لذا تستخدم ECC `ecc@ecc` للحفاظ على أسماء الأدوات وأسماء الأوامر القصيرة ضمن قيود التحقق الصارمة لسطح المكتب/واجهة برمجة التطبيقات. قد تعرض المشاركات القديمة الاسم الطويل الأقدم؛ اعتبر ذلك مجرد اسم مستعار قديم. منفصلًا عن ذلك، بقيت حزمة npm على `ecc-universal`، لذا فإن تثبيتات npm وتثبيتات السوق تختلف عمدًا.

### الخطوة 2: ثبّت القواعد فقط إذا كنت تحتاجها

> تحذير: **مهم:** لا تستطيع إضافات Claude Code توزيع `rules` تلقائيًا.
>
> إذا كنت قد ثبتت ECC بالفعل عبر `/plugin install`، **لا تشغّل** `./install.sh --profile full` أو `.\install.ps1 --profile full` أو `npx ecc-install --profile full` بعد ذلك. تقوم الإضافة بتحميل مهارات ECC وأوامره وخطافاته بالفعل. تشغيل المثبت الكامل بعد تثبيت الإضافة ينسخ نفس الأسطح إلى دلائل المستخدم ويستطيع أن ينشئ مهارات ومهام تشغيل مكررة.
>
> بالنسبة لتثبيت الإضافة، انسخ يدويًا فقط مجلدات `rules/` التي تريدها تحت `~/.claude/rules/ecc/`. ابدأ بـ `rules/common` بالإضافة إلى لغة أو حزمة إطار واحد تستخدمها بالفعل. لا تنسخ كل مجلدات القواعد ما لم تكن ترغب صراحة في كل ذلك السياق داخل Claude.

```bash
# استنخِ المستودع أولاً
git clone https://github.com/affaan-m/ECC.git
cd ECC

# ثبّت التبعيات (اختر مدير الحزم)
npm install        # أو: pnpm install | yarn install | bun install

# مسار تثبيت الإضافة: انسخ فقط قواعد ECC إلى مساحة ECC الملكية
mkdir -p ~/.claude/rules/ecc
cp -R rules/common ~/.claude/rules/ecc/
cp -R rules/typescript ~/.claude/rules/ecc/

# مسار تثبيت ECC اليدوي الكامل (استخدم هذا بدلًا من /plugin install)
# ./install.sh --profile full
```

```powershell
# Windows PowerShell

# مسار تثبيت الإضافة: انسخ فقط قواعد ECC إلى مساحة ECC الملكية
New-Item -ItemType Directory -Force -Path "$HOME/.claude/rules/ecc" | Out-Null
Copy-Item -Recurse rules/common "$HOME/.claude/rules/ecc/"
Copy-Item -Recurse rules/typescript "$HOME/.claude/rules/ecc/"

# مسار تثبيت ECC اليدوي الكامل (استخدم هذا بدلًا من /plugin install)
# .\install.ps1 --profile full
# npx ecc-install --profile full
```

للاطلاع على تعليمات التثبيت اليدوي، راجع README في المجلد `rules/`. عند نسخ القواعد يدويًا، انسخ الدليل اللغوي الكامل (على سبيل المثال `rules/common` أو `rules/golang`)، وليس الملفات بداخله، حتى تظل المراجع النسبية تعمل ولا تتصادم الأسماء.

### تثبيت الخطافات

لا تنسخ ملف `hooks/hooks.json` الخام إلى `~/.claude/settings.json` أو `~/.claude/hooks/hooks.json`. هذا الملف خاص بالإضافة/المستودع ومُعد ليتم تثبيته عبر مثبت ECC أو تحميله كمكوّن إضافي، لذا النسخ الخام ليس مسارًا يدعم.

استخدم المثبت لكتثبيت وقت تشغيل خطاف Claude فقط حتى يتم إعادة كتابة مسارات الأوامر بشكل صحيح:

```bash
# macOS / Linux
bash ./install.sh --target claude --modules hooks-runtime
```

```powershell
# Windows PowerShell
pwsh -File .\install.ps1 --target claude --modules hooks-runtime
```

هذا يكتب ملفات الخطاف المحلولة إلى `~/.claude/hooks/hooks.json` ويترك أي `~/.claude/settings.json` موجود بدون تغيير.

إذا قمت بتثبيت ECC عبر `/plugin install`، فلا تنسخ تلك الخطافات إلى `settings.json`. يقوم Claude Code v2.1+ بتحميل `hooks/hooks.json` من الإضافة تلقائيًا، ونسخها المكرر إلى `settings.json` يسبب تنفيذًا مكررًا ومشاكل عبر الأنظمة.

ملاحظة Windows: دليل تكوين Claude هو `%USERPROFILE%\\.claude`، وليس `~/claude`.

#### تهيئة MCPs

لا تقوم تثبيتات إضافات Claude تلقائيًا بتمكين تعريفات خادم MCP المجمعة مع ECC. هذا يتجنب أسماء أدوات MCP الطويلة للغاية على بوابات الجهات الخارجية الصارمة بينما يحتفظ بإعداد MCP اليدوي متاحًا.

استخدم أمر `/mcp` في Claude Code أو إعداد MCP المدار من CLI لتغييرات خادم Claude Code الحية. استخدم `/mcp` لتعطيل وقت التشغيل؛ يقوم Claude Code بتخزين تلك الخيارات في `~/.claude.json`.

للوصول المحلي إلى MCP في المستودع، انسخ تعريفات خادم MCP المطلوبة من `mcp-configs/mcp-servers.json` إلى `.mcp.json` على مستوى المشروع.

يأتي ECC بموصل افتراضي واحد فقط (`chrome-devtools`); كل شيء آخر هو مهارة تغلف CLI/REST API أو إدخال كتالوج اختياري. يعيش القاعدة ومراجعة يونيو 2026 التي ألغت الافتراءات السابقة في [docs/MCP-CONNECTOR-POLICY.md](docs/MCP-CONNECTOR-POLICY.md).

إذا كنت تدير نسخك الخاصة من خوادم MCP الخاصة بـ ECC، قم بتعيين:

```bash
export ECC_DISABLED_MCPS="chrome-devtools"
```

يقوم تثبيت ECC المدار وتدفقات المزامنة بتخطي أو إزالة هذه الخوادم المجمعة بدلاً من إعادة إضافتها المكررة. `ECC_DISABLED_MCPS` هو مرشح تثبيت/مزامنة ECC، وليس تبديلًا مباشرًا في Claude Code.

**مهم:** استبدل عناصر العنونة `YOUR_*_HERE` بمفاتيح API الحقيقية.

## المفاهيم الرئيسية

### الوكلاء

الوكلاء الفرعيون يتعاملون مع المهام المفصلة ذات النطاق المحدود. مثال:

```markdown
---
name: code-reviewer
description: Reviews code for quality, security, and maintainability
tools: ["Read", "Grep", "Glob", "Bash"]
model: opus
---

You are a senior code reviewer...
```

### المهارات

المهارات هي السطح الأساسي لتدفقات العمل. يمكن استدعاؤها مباشرة، اقتراحها تلقائيًا، وإعادة استخدامها. لا تزال ECC تقدم أوامر مدعومة أثناء الانتقال، بينما تبقى اختصارات الأسماء القصيرة المتقادمة ضمن `legacy-command-shims/` للاختيار الصريح فقط. يجب أن يتم تطوير مهارات جديدة في `skills/` أولًا.

```markdown
# TDD Workflow

1. Define interfaces first
2. Write failing tests (RED)
3. Implement minimal code (GREEN)
4. Refactor (IMPROVE)
5. Verify 80%+ coverage
```

### الخطافات

الخطافات تُفعَّل عند أحداث الأداة. مثال - تحذير عند وجود console.log:

```json
{
  "matcher": "tool == \"Edit\" && tool_input.file_path matches \"\\.(ts|tsx|js|jsx)$\"",
  "hooks": [{
    "type": "command",
    "command": "#!/bin/bash\ngrep -n 'console\\.log' \"$file_path\" && echo '[Hook] Remove console.log' >&2"
  }]
}
```

### القواعد

القواعد هي إرشادات يجب اتباعها دائمًا، منظمة إلى `common/` (مبادئ عامة للغة) + أدلة خاصة بكل لغة:

```
rules/
  common/          # Universal principles (always install)
  typescript/      # TS/JS specific patterns and tools
  python/          # Python specific patterns and tools
  golang/          # Go specific patterns and tools
  swift/           # Swift specific patterns and tools
  php/             # PHP specific patterns and tools
  arkts/           # HarmonyOS / ArkTS patterns and constraints
```

راجع [`rules/README.md`](rules/README.md) لتعليمات التثبيت وبنية المشروع.

## أي وكيل يجب أن أستخدم؟

لست متأكدًا من أين تبدأ؟ استخدم هذا المرجع السريع. المهارات هي السطح الأساسي لتدفق العمل؛ الأوامر المدعومة تبقى متاحة للتدفقات القائمة على الأوامر.

| أريد أن... | استخدم هذا السطح | الوكيل المستخدم |
|--------------|-----------------|------------|
| خطط ميزة جديدة | `/ecc:plan "Add auth"` | planner |
| صمّم هندسة النظام | `/ecc:plan` + وكيل architect | architect |
| اكتب شفرة مع اختبارات أولًا | مهارة `tdd-workflow` | tdd-guide |
| راجع الشفرة التي كتبتها للتو | `/code-review` | code-reviewer |
| أصلح بناء فاشل | `/build-fix` | build-error-resolver |
| شغل اختبارات نهاية إلى نهاية | مهارة `e2e-testing` | e2e-runner |
| اكتشف ثغرات أمنية | `/security-scan` | security-reviewer |
| أزل الشفرة الميتة | `/refactor-clean` | refactor-cleaner |
| حدّث المستندات | `/update-docs` | doc-updater |
| راجع شفرة Go | `/go-review` | go-reviewer |
| راجع شفرة Python | `/python-review` | python-reviewer |
| راجع شفرة F# | *(استخدم `fsharp-reviewer` مباشرة)* | fsharp-reviewer |
| راجع TypeScript/JavaScript | *(استخدم `typescript-reviewer` مباشرة)* | typescript-reviewer |
| طوّر تطبيقات HarmonyOS | *(استخدم `harmonyos-app-resolver` مباشرة)* | harmonyos-app-resolver |
| راقب استعلامات قاعدة البيانات | *(تفويض تلقائي)* | database-reviewer |
| راجع تغييرات ML الإنتاجية | مهارة `mle-workflow` + وكيل `mle-reviewer` | mle-reviewer |

### تدفقات العمل الشائعة

النماذج التالية تظهر حيث تظل الأسماء القصيرة جزءًا من السطح المدعوم. الاختصارات القديمة مثل `/tdd` و `/eval` تعيش في `legacy-command-shims/` فقط إذا اخترتها صراحة.

**بدء ميزة جديدة:**
```
/ecc:plan "Add user authentication with OAuth"
                                              → planner creates implementation blueprint
tdd-workflow skill                            → tdd-guide enforces write-tests-first
/code-review                                  → code-reviewer checks your work
```

**إصلاح علة:**
```
tdd-workflow skill                            → tdd-guide: write a failing test that reproduces it
                                              → implement the fix, verify test passes
/code-review                                  → code-reviewer: catch regressions
```

**التحضير للإنتاج:**
```
/security-scan                                → security-reviewer: OWASP Top 10 audit
e2e-testing skill                             → e2e-runner: critical user flow tests
/test-coverage                                → verify 80%+ coverage
```

<details>
<summary><b>هل يمكنني استخدام ECC مع Claude Code عبر نقطة نهاية API مخصصة أو بوابة نموذج؟</b></summary>

نعم. ECC لا يثبّت إعدادات ناقل Anthropic. يعمل محليًا عبر واجهة Claude Code العادية، لذا يعمل مع:

- Claude Code المستضاف من Anthropic
- إعدادات بوابة Claude Code الرسمية باستخدام `ANTHROPIC_BASE_URL` و `ANTHROPIC_AUTH_TOKEN`
- نقاط نهاية متوافقة مع Anthropic API التي يتوقعها Claude Code

مثال بسيط:

```bash
export ANTHROPIC_BASE_URL=https://your-gateway.example.com
export ANTHROPIC_AUTH_TOKEN=your-token
claude
```

إذا أعادت بوابتك تسمية أسماء النماذج، فقم بتكوين ذلك في Claude Code بدلاً من ECC. أصبحت خطافات ECC، المهارات، الأوامر، والقواعد مستقلة عن موفر النموذج بمجرد عمل CLI `claude`.

المراجع الرسمية:
- [Claude Code LLM gateway docs](https://docs.anthropic.com/en/docs/claude-code/llm-gateway)
- [Claude Code model configuration docs](https://docs.anthropic.com/en/docs/claude-code/model-config)

</details>

<details>
<summary><b>سياق النافذة ينقص / Claude ينفد من السياق</b></summary>

الكثير من خوادم MCP تستهلك السياق. كل وصف أداة MCP يستخدم رموزًا من نافذة 200k، مما قد يقللها إلى ~70k. يتم تحديد سياق SessionStart الإضافي إلى 8000 حرف افتراضيًا؛ قلله إلى `ECC_SESSION_START_MAX_CHARS=4000` أو عطّله بـ `ECC_SESSION_START_CONTEXT=off` للإعدادات المحلية النموذج أو منخفضة السياق.

**الإصلاح:** عطل خوادم MCP غير المستخدمة من Claude Code باستخدام `/mcp`. يقوم Claude Code بتخزين تلك الخيارات في `~/.claude.json`; `.claude/settings.json` و `.claude/settings.local.json` ليست موثوقة للتبديل بعد التحميل.

احتفظ بأقل من 10 MCPs مفعلة و80 أداة نشطة.
</details>

<details>
<summary><b>هل يعمل هذا مع Cursor / OpenCode / Codex / Antigravity / GitHub Copilot؟</b></summary>

- **Cursor**: تكوينات مترجمة مسبقًا في `.cursor/`. راجع [Cursor IDE Support](#cursor-ide-support).

- **JoyCode / CodeBuddy**: محولات تثبيت اختياري لمهام الأوامر، الوكلاء، المهارات، والقواعد المسطحة.
- **Qwen CLI**: محول تثبيت منزلي لمهام الأوامر، الوكلاء، المهارات، القواعد، وتكوين Qwen.
- **Zed**: محول تثبيت مشروع محلي لـ `.zed/settings.json`، القواعد المسطحة، الأوامر، الوكلاء، والمهارات.
- **الحوامل غير الأصلية**: مسار تسقيفي يدوي لـ Grok وما شابه. راجع [Manual Adaptation Guide](docs/MANUAL-ADAPTATION-GUIDE.md).
- **Claude Code**: الأصل — الهدف الأساسي.

## Token Optimization

Claude Code usage can be expensive if you don't manage token consumption. These settings significantly reduce costs without sacrificing quality.

### Recommended Settings

Add to `~/.claude/settings.json`:

```json
{
  "model": "sonnet",
  "env": {
    "MAX_THINKING_TOKENS": "10000",
    "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "50"
  }
}
```

| Setting | Default | Recommended | Impact |
|---------|---------|-------------|--------|
| `model` | opus | **sonnet** | ~60% cost reduction; handles 80%+ of coding tasks |
| `MAX_THINKING_TOKENS` | 31,999 | **10,000** | ~70% reduction in hidden thinking cost per request |
| `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` | 95 | **50** | Compacts earlier — better quality in long sessions |
| `ECC_CONTEXT_MONITOR_COST_WARNINGS` | on | **off for subscription users** | Suppresses agent-facing API-rate estimate warnings while keeping context/scope/loop warnings |

Switch to Opus only when you need deep architectural reasoning:
```
/model opus
```

### Daily Workflow Commands

| Command | When to Use |
|---------|-------------|
| `/model sonnet` | Default for most tasks |
| `/model opus` | Complex architecture, debugging, deep reasoning |
| `/clear` | Between unrelated tasks (free, instant reset) |
| `/compact` | At logical task breakpoints (research done, milestone complete) |
| `/cost` | Monitor token spending during session |

If you use a Claude subscription and the context monitor's API-rate estimates are not useful, set `ECC_CONTEXT_MONITOR_COST_WARNINGS=off`. This only suppresses the agent-facing cost warnings; it does not disable context exhaustion, scope, or loop warnings.

### Strategic Compaction

The `strategic-compact` skill (included in this plugin) suggests `/compact` at logical breakpoints instead of relying on auto-compaction at 95% context. See `skills/strategic-compact/SKILL.md` for the full decision guide.

**When to compact:**
- After research/exploration, before implementation
- After completing a milestone, before starting the next
- After debugging, before continuing feature work

**When NOT to compact:**
- Mid-implementation (you'll lose variable names, file paths, partial state)

### Context Window Management

**Critical:** Don't enable all MCPs at once. Each MCP tool description consumes tokens from your 200k window, potentially reducing it to ~70k.

- Keep under 10 MCPs enabled per project
- Keep under 80 tools active
- Use `/mcp` to disable unused Claude Code MCP servers; those runtime choices persist in `~/.claude.json`
- Use `ECC_DISABLED_MCPS` only to filter ECC-generated MCP configs during install/sync flows

### Agent Teams Cost Warning

Agent Teams spawns multiple context windows. Each teammate consumes tokens independently. Only use for tasks where parallelism provides clear value (multi-module work, parallel reviews). For simple sequential tasks, subagents are more token-efficient.
