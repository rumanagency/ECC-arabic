# الدرس الرابع: دمج النظام مع المشاريع الحالية والتطبيقات العملية

  الدرس الرابع: دمج النظام مع المشاريع الحالية والتطبيقات العملية  

## حالة استخدام: تنظيف وإصلاح تطبيق Express قديم

تخيل أنك انضممت إلى فريق يعمل على تطبيق Node.js / Express موجود منذ 4 سنوات. الكود مليء بالملفات المهجورة، الأمان ضعيف، والكود غير مرتب.

إدخال جميع وكلاء وخطافات ECC دفعة واحدة قد يؤدي لكسر المشروع بالكامل. لذلك، يجب البدء بعملية الدمج **الانتقائي والتدريجي**.

### الخطوة الأولى: التثبيت الانتقائي (Selective Install)

لا تقم بتثبيت الملف الشخصي الكامل (Full Profile). بدلاً من ذلك، سنستشير ECC لمعرفة ما نحتاجه فقط لهذا المشروع القديم:

```
# أولاً نطلب نصيحة النظام بناءً على احتياجنا
npx ecc consult "refactoring and security reviews for legacy express js app" --target claude

# بناءً على النصيحة، سنقوم بنسخ وكيلين فقط بشكل يدوي إلى مساحة العمل الحالية لتجنب التأثير على باقي المشروع
mkdir -p .claude/agents
cp ~/.claude/agents/refactor-cleaner.md .claude/agents/
cp ~/.claude/agents/security-reviewer.md .claude/agents/
```

## الخطوة الثانية: إنشاء خرائط الكود لتفادي استهلاك السياق (Context)

في التطبيقات القديمة، قراءة جميع الملفات لمحاولة الفهم ستؤدي لاستهلاك الـ 200k رموز بسرعة وتكلفك الكثير.

قم باستخدام أداة البحث المتقدمة `mgrep` (إذا قمت بتثبيتها عبر إضافات Claude) أو أداة `grep` الأساسية لفهرسة الملفات المهمة فقط دون فتحها:

```
/mgrep "app.post|app.get|require('express')"
```

سيجد النظام نقاط الدخول الأساسية ويوجه الوكلاء لقراءتها فقط، وبالتالي تقليل سياق القراءة بشكل هائل.

## الخطوة الثالثة: التنظيف الآمن (Safe Refactoring)

لدينا مجلد مليء بالوظائف المهجورة. دعونا نشغّل الوكيل الخاص بالتنظيف **على ملف واحد فقط** لتجربته:

```
# توجيه الوكيل للملف وتحديد مهمته
/refactor-clean "قم بتنظيف ملف `src/routes/auth.js` وإزالة كافة الوظائف والمتغيرات التي لم تعد تُستخدم في المشروع."
```

**ماذا سيفعل `refactor-cleaner`؟**

1.  سيقوم بمسح المتغيرات الميتة (Dead Variables).
2.  لن يغير المنطق الأساسي للبرنامج، لأنه مكلف بمهمة محددة النطاق.
3.  قد يستخدم أدوات `git diff` ليعرض لك التغييرات.

## الخطوة الرابعة: المراجعة الأمنية والمراقبة (Security Audit)

بعد التنظيف، دعونا نرى ما إذا كان الكود القديم يحتوي على ثغرات عبر وكيل `security-reviewer`:

```
/security-scan "قم بتحليل الثغرات الأمنية (OWASP Top 10) داخل مجلد `src/routes/` وتأكد من منع هجمات حقن قاعدة البيانات (SQL Injection)."
```

هذا يضمن أن الكود القديم، الذي ربما لم يتم فحصه منذ سنوات، سيخضع لمعايير أمنية قوية ومستقلة دون إرهاق الوكيل الرئيسي (Claude).

## التراجع الآمن باستخدام Worktrees

بما أنك تقوم بتعديلات جوهرية على نظام موجود، استخدم بيئات آمنة تماماً:

```
# أنشئ شجرة عمل جديدة للـ Refactoring دون المساس بالفرع الرئيسي
git worktree add ../legacy-refactor legacy-refactor-branch
cd ../legacy-refactor
```

الآن، مهما ارتكب الذكاء الاصطناعي من أخطاء في التعديل والحذف، فالبيئة الأساسية الخاصة بك بأمان ولن تتأثر التغييرات إلا إذا قمت بدمجها بنفسك.
