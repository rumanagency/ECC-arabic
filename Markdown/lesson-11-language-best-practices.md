# الدرس الحادي عشر: أفضل ممارسات اللغات البرمجية مع ECC

  الدرس الحادي عشر: أفضل ممارسات اللغات البرمجية مع ECC  

## مشكلة النماذج العامة (General AI Models)

بشكل افتراضي، عند طلب كود Python من Claude، قد يكتبه بأسلوب يشبه Java. وعند طلب كود React، قد يستخدم كوداً قديماً. **الحل؟** نظام "القواعد" (Rules) في ECC والوكلاء المتخصصون.

## 1\. استخدام مجلد القواعد (Rules Directory)

يحتوي نظام ECC على مجلد قواعد ضخم مقسم حسب اللغات وأطر العمل (مثل: `rules/python`, `rules/go`, `rules/rust`). يجب عليك دائماً نسخ القواعد الخاصة بمشروعك محلياً لفرضها.

### مثال: مشروع لغة Go

```
# توجيه Claude لفهم أن هذا مشروع Go
mkdir -p .claude/rules/ecc
cp -R ~/.claude/rules/common .claude/rules/ecc/
cp -R ~/.claude/rules/go .claude/rules/ecc/
```

بمجرد فعل ذلك، سيبدأ Claude تلقائياً بفهم أن دوال Go يجب أن ترجع الأخطاء (Errors) بشكل صريح، وأنه يجب استخدام الـ Goroutines بفعالية، ولن يكتب كود Go بأسلوب لغة أخرى.

## 2\. الاستعانة بالمراجعين المتخصصين (Specialized Reviewers)

بعد كتابة الكود أو إضافة ميزة عبر `tdd-workflow`، لا تستخدم أمر المراجعة العام. يوفر ECC وكلاء فرعيين متخصصين لكل لغة تقريباً لمراجعة الكود!

### حالة استخدام: مراجعة مشروع Django / Python

لديك ميزة جديدة أضفتها لتطبيق Django. كيف تتأكد أنها تتبع معايير PEP8 وأنك لا تعاني من مشكلة استعلامات الـ N+1 في الـ ORM؟

```
/django-reviewer "قم بمراجعة ملف \`models.py\` وملف \`views.py\` الجدد. تأكد من عدم وجود استعلامات N+1 ومن الالتزام بأفضل ممارسات Django REST Framework."
```

الوكيل `django-reviewer` مبرمج مسبقاً بقاعدة معرفية عميقة حول إطار العمل، وسيقوم بالتنبيه على تفاصيل دقيقة جداً لا يلاحظها المراجع العادي.

## 3\. حل مشاكل البناء اللغوية (Build Resolvers)

أحياناً تواجه أخطاء ترجمة (Compilation Errors) معقدة في لغات مثل Rust أو C++. يوفر ECC وكلاء مصممين خصيصاً لتحليل رسائل الخطأ الغامضة:

```
# بدلاً من البحث في جوجل لساعات:
/rust-build-resolver "تحليل سجل أخطاء الـ Cargo المرفق وحل مشاكل الـ Borrow Checker في ملف \`main.rs\`."
```

يقوم الوكيل بقراءة رسالة خطأ المترجم (Compiler) وتصحيح استخدام الذاكرة وفقاً لأسلوب لغة Rust الصارم.

## ملخص القواعد الذهبية

1.  انقل دائماً `rules/<language>` إلى مسارك المحلي.
2.  استخدم المراجع المتخصص للغتك (مثل `go-reviewer`, `kotlin-reviewer`).
3.  استخدم محلل أخطاء البناء الخاص ببيئتك (مثل `java-build-resolver`).
