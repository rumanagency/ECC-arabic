# الدرس الثالث: المشاريع الجديدة مع ECC والتطبيقات العملية

  الدرس الثالث: المشاريع الجديدة مع ECC والتطبيقات العملية  

## حالة استخدام عملية: بناء مدونة باستخدام Next.js 15

تخيل أنك تود البدء بمشروع جديد لبناء مدونة تقنية باستخدام Next.js. إذا استخدمت المنهج التقليدي (كتابة الكود فوراً)، ستحصل على كود غير منظم وقديم في الغالب. إليك الطريقة الصحيحة باستخدام نظام ECC.

### الخطوة الأولى: إعداد الهيكل الأساسي

قم بإنشاء المشروع عبر الأوامر القياسية الخاصة بـ Next.js أولاً داخل الطرفية (Terminal):

```
npx create-next-app@latest my-tech-blog --typescript --tailwind --eslint
cd my-tech-blog
```

### الخطوة الثانية: تجهيز قواعد (Rules) لغة المشروع

بما أن المشروع يستخدم TypeScript و React، يجب أن نضمن أن وكلاء الذكاء الاصطناعي يفهمون أفضل الممارسات الخاصة بها.

```
# قم بنسخ قواعد ECC العالمية وقواعد TypeScript إلى مجلد مشروعك
mkdir -p ./.claude/rules/ecc
cp -R ~/.claude/rules/common ./.claude/rules/ecc/
cp -R ~/.claude/rules/typescript ./.claude/rules/ecc/
```

**لماذا هذه الخطوة حرجة؟** بدون هذه القواعد، قد يقترح Claude استخدام أنماط قديمة (مثل Class Components في React) أو يتجاهل قواعد كتابة TypeScript الصارمة.

## الخطوة الثالثة: التخطيط الدقيق مع الوكيل المخطط (Planner)

الآن، افتح واجهة محادثة Claude Code في مسار المشروع، واطلب التخطيط **قبل كتابة أي كود**:

```
/ecc:plan "أريد بناء مدونة تقنية باستخدام Next.js App Router، تدعم الوضع الداكن (Dark Mode) وتستخدم Markdown للمقالات. قم بتجهيز هيكل الملفات والمكونات اللازمة."
```

**ماذا سيحدث عملياً؟**

*   سيقوم وكيل `planner` باستدعاء وكيل النظام `architect` (إذا تطلب الأمر) لتقرير كيفية تخزين المقالات.
*   لن يقوم الوكيل بكتابة كود الإنتاج أبداً. بل سينشئ ملف خطة تنفيذي شامل (غالباً بصيغة Markdown) يضع فيه خارطة طريق مفصلة.

## الخطوة الرابعة: التطوير المرحلي الموجه بالاختبار (TDD)

الآن وقد أصبح لديك خطة واضحة، حان وقت التنفيذ. بدلاً من الطلب منه برمجة كل شيء دفعة واحدة، سنطلب منه تنفيذ أول خطوة من الخطة باستخدام مهارة الـ TDD لضمان الموثوقية:

```
tdd-workflow skill "قم بتنفيذ الخطوة الأولى من الخطة وهي إنشاء مكون رأس الصفحة (Header Component) مع دعم التبديل للوضع الداكن."
```

**سيناريو التنفيذ الواقعي (RED -> GREEN -> REFACTOR):**

1.  **(RED):** سيكتب الوكيل `tdd-guide` ملف اختبار مثل `Header.test.tsx` يتحقق من وجود زر الوضع الداكن. بطبيعة الحال سيفشل هذا الاختبار لأن المكون غير موجود بعد.
2.  **(GREEN):** سيقوم الوكيل الآن بكتابة الكود الفعلي `Header.tsx` لضمان نجاح الاختبار المكتوب مسبقاً.
3.  **(REFACTOR):** أخيراً، سيقوم بتحسين الكود وتنسيقه دون تغيير وظيفته لتلبية معايير الجودة بنسبة 80% على الأقل لتغطية الاختبارات.

## حالة استخدام متقدمة: بناء منصة SaaS (Laravel + Next.js + Postgres)

لنفترض أنك تبني مشروعاً ضخماً يتكون من واجهة خلفية (Backend) بـ Laravel، وواجهة أمامية (Frontend) بـ Next.js، وقاعدة بيانات PostgreSQL. كيف نجهز هذه البيئة المعقدة مع ECC؟

### 1\. هيكلة المستودع (Monorepo)

```
mkdir my-saas-project && cd my-saas-project
# إنشاء الواجهة الأمامية والخلفية
npx create-next-app@latest frontend
composer create-project laravel/laravel backend
```

### 2\. استراتيجية توزيع القواعد (Rules Strategy)

في المستودعات الأحادية (Monorepos) التي تحتوي على لغات برمجة مختلفة، لدينا خياران استراتيجيان لإدارة قواعد الذكاء الاصطناعي (Claude Rules):

#### الخيار الأول: العزل التام (مجلدان `.claude` منفصلان) - الموصى به للمشاريع الضخمة

قمنا بإنشاء مجلد `.claude` في الـ Frontend وآخر في الـ Backend. **لماذا؟**

*   **توفير السياق (Context Window):** عندما تقوم بتشغيل `claude` داخل مجلد `frontend`، سيقرأ قواعد TypeScript فقط. لن يقرأ قواعد PHP مطلقاً، مما يوفر آلاف الرموز (Tokens) ويقلل التكلفة.
*   **منع التشتت:** يمنع Claude من تطبيق قواعد تصميم خاصة بـ Laravel بالخطأ على كود React.

```
# للواجهة الأمامية
mkdir -p frontend/.claude/rules/ecc
cp -R ~/.claude/rules/common frontend/.claude/rules/ecc/
cp -R ~/.claude/rules/typescript frontend/.claude/rules/ecc/

# للواجهة الخلفية
mkdir -p backend/.claude/rules/ecc
cp -R ~/.claude/rules/common backend/.claude/rules/ecc/
cp -R ~/.claude/rules/php backend/.claude/rules/ecc/
```

#### الخيار الثاني: مجلد `.claude` واحد في الجذر (Root)

ماذا لو وضعنا المجلد في الجذر `my-saas-project/.claude` ونسخنا كافة القواعد (TypeScript و PHP) داخله؟

*   **الميزة:** يمكنك تشغيل أمر `claude` من مسار الجذر، وتطلب منه بناء واجهة في الـ Frontend وإنشاء الـ API الخاص بها في الـ Backend **في أمر واحد** (Full-stack Prompting).
*   **العيب:** سيقوم Claude بتحميل كافة قواعد اللغتين في كل رسالة ترسلها، مما يزيد استهلاك مساحة السياق بسرعة.
*   **كيف تنجح بها؟** إذا اخترت هذا النهج، يفضل عدم وضع الكثير من ملفات القواعد المفصلة في الجذر، والاعتماد بدلاً من ذلك على استدعاء **المهارات (Skills)** بشكل صريح عندما تحتاجها لتقليل العبء على النظام.

### 3\. إعداد الاتصال بقاعدة البيانات (Postgres MCP)

لكي يستطيع Claude قراءة مخططات قاعدة البيانات (Schemas) وكتابة استعلامات دقيقة، يجب تفعيل خادم Postgres MCP في الجذر:

```
// إنشاء ملف .mcp.json في جذر المشروع (my-saas-project)
{
  "postgres": {
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-postgres", "postgresql://user:password@localhost:5432/saas_db"]
  }
}
```

### 4\. استخدام Hookify للأتمتة

نريد التأكد من تنسيق الكود تلقائياً (Prettier للـ Frontend و Pint للـ Backend). بدلاً من كتابة JSON معقد، نستخدم الإضافة `hookify`:

```
# في طرفية Claude
/hookify "أنشئ خطافاً يعمل بعد تحرير أي ملف php في مجلد backend ليقوم بتشغيل أمر './vendor/bin/pint'"
/hookify "أنشئ خطافاً يعمل بعد تحرير أي ملف ts/tsx في مجلد frontend ليقوم بتشغيل أمر 'npx prettier --write'"
```

### 5\. التخطيط المعماري الشامل

الآن بيئتك جاهزة تماماً. ابدأ بسؤال الوكيل المعماري (Architect) لربط النظامين معاً:

```
/ecc:plan "قم بتخطيط بنية المصادقة (Authentication) بين Next.js و Laravel باستخدام Laravel Sanctum. حدد مسارات الـ API وشكل استجابات JSON."
```
