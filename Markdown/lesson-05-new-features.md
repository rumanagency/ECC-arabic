# الدرس الخامس: بناء ميزات جديدة مع ECC والتطبيقات العملية

  الدرس الخامس: بناء ميزات جديدة مع ECC والتطبيقات العملية  

## حالة استخدام عملية: إضافة ميزة تسجيل الدخول (OAuth Login)

نريد إضافة ميزة تسجيل الدخول للمستخدمين عبر GitHub. لن نطلب من Claude أن يكتب الكود فوراً لئلا نحصل على كود غير مغطى باختبارات أو مليء بالثغرات.

### الخطوة الأولى: التخطيط الصارم (Planning)

```
/ecc:plan "أريد إضافة ميزة تسجيل الدخول عبر GitHub باستخدام Passport.js. قم بوضع الهيكل الأساسي وتحديد الملفات التي سيتم إنشاؤها وتعديلها."
```

**لماذا نبدأ هنا؟** الوكيل `planner` سيولد وثيقة Markdown واضحة، ليراجعها المستخدم. إذا كانت الخطة جيدة، نمضي للخطوة الثانية.

## الخطوة الثانية: دورة الـ TDD الفعلية

يفرض نظام ECC تغطية اختبارات بنسبة 80% على الأقل. سنستخدم مهارة \`tdd-workflow\` ووكيل \`tdd-guide\` لبناء ميزة تسجيل الدخول خطوة بخطوة.

```
tdd-workflow skill "بناءً على الخطة، قم بإنشاء مسار الـ API (Route) الخاص بتسجيل دخول GitHub."
```

### المرحلة الأولى: الفشل المتعمد (RED)

سيكتب الوكيل ملف اختبار (مثلاً `auth.test.js`) يتحقق من أن المسار `/auth/github` يرد بالرمز 302 (إعادة توجيه). عند تشغيل الاختبار سيظهر لون **أحمر** لأن الكود الفعلي لم يُكتب بعد.

### المرحلة الثانية: اجتياز الاختبار (GREEN)

سيقوم الوكيل الآن بكتابة **الحد الأدنى** من كود الإنتاج (مثلاً في `auth.routes.js`) لجعل مسار الـ API يعمل بشكل صحيح بحيث ينجح الاختبار ويتحول اللون إلى **أخضر**.

### المرحلة الثالثة: التحسين (REFACTOR)

سيقوم الوكيل بتحسين كود `auth.routes.js` (إضافة التعليقات، ترتيب الاستيرادات، تطبيق أفضل الممارسات)، وفي كل خطوة يتأكد أن الاختبار ما زال يعطي لوناً أخضر.

## الخطوة الثالثة: المراجعة التلقائية (Code Review)

بعد انتهاء الـ TDD بنجاح، لدينا كود يعمل ومغطى بالاختبارات. ولكن هل الكود نظيف وقابل للصيانة؟ نمرره للوكيل `code-reviewer`:

```
/code-review "قم بمراجعة الكود الذي تمت إضافته مؤخراً لمسارات المصادقة (Auth Routes). هل يتوافق مع معايير المشروع؟"
```

قد يوصي الوكيل بفصل منطق التحكم (Controller) عن ملف الـ Routes، مما يجعل التطبيق أكثر تنظيماً.

## الخطوة الرابعة: المراجعة الأمنية الصارمة (Security Review)

بما أننا نتعامل مع عملية تسجيل الدخول، فإن الأمان يمثل أولوية قصوى. سنقوم بمسح أمني فوراً باستخدام `security-reviewer`:

```
/security-scan "قم بتحليل ملفات المصادقة الجديدة بحثاً عن أي ثغرات أو أسرار (Secrets) تم رفعها بالخطأ."
```

**ماذا نتوقع؟**

*   إذا قمنا بكتابة `GITHUB_CLIENT_SECRET="12345"` داخل الكود بدلاً من ملف `.env`، سيتوقف الوكيل فوراً ويطلق تحذيراً لتغيير هذا السلوك.
*   سيتأكد الوكيل من وجود حماية ضد هجمات تزوير الطلبات (CSRF).
